## Problem Statement

현재 인메모리 기반 미니 DBMS는 로컬 SQL 실행기 형태로는 동작하지만, 외부 클라이언트가 HTTP를 통해 안전하게 요청을 보내고 결과를 JSON으로 받는 서버 인터페이스는 없다.

이번 설계의 핵심 문제는 "새로운 DB를 만드는 것"이 아니라, 이미 있는 `sql.c`, `table.c`, `bptree.c`를 최대한 유지하면서 다음 세 가지를 동시에 만족시키는 서버 층을 붙이는 것이다.

- 여러 클라이언트 요청을 스레드 풀로 처리할 것
- 쓰기 요청은 직렬화해서 데이터 불일치를 막을 것
- 읽기 요청은 MVP 범위 안에서 가능한 한 단순하게 병렬 처리할 것

즉, 이 문서는 기존 미니 DBMS를 "설명 가능한 병렬 API 서버"로 감싸는 가장 작은 설계를 정의한다.

## What Makes This Cool

- 이미 만든 SQL 엔진과 B+Tree 인덱스를 버리지 않고 그대로 재사용한다.
- 단순한 CRUD 서버가 아니라, 큐 포화, lock timeout, `usedIndex`, metrics 같은 시스템 관점의 신호를 API 계약에 직접 드러낸다.
- 발표와 데모에서 `client -> queue -> thread pool -> sql_execute -> table/B+Tree` 흐름을 눈으로 보여주기 좋다.
- 완전한 DB 수준의 정합성보다 "현재 범위에서 무엇을 보장하고 무엇을 포기하는지"를 명확하게 설명하는 설계라서 팀원 간 공유와 구현 분담이 쉽다.

## Constraints

- 기존 `sql.c`, `table.c`, `bptree.c`의 public API는 가능한 유지한다.
- 지원 SQL은 현재 엔진이 처리 가능한 `INSERT`, `SELECT`, 단일 `users` 테이블 범위로 제한한다.
- 외부 JSON 라이브러리를 도입하지 않는다.
- API는 `GET /health`, `GET /metrics`, `POST /query`만 제공한다.
- write lock timeout은 "전체 요청 시간"이 아니라 "write mutex 획득 대기 시간"에만 적용한다.
- `SELECT`는 이번 MVP에서 table lock 없이 실행한다.
- race-free read/write 보장은 이번 단계의 목표가 아니며, 문서화와 데모 설명으로 관리한다.

## Premises

- 기존 SQL 엔진은 서버 프로세스 안에서 재사용해도 충분히 안정적이다.
- `WHERE id ...` 조회는 계속 B+Tree 경로를 사용하므로, API 레이어에서 `usedIndex`를 단순 판정해도 사용자 기대와 크게 어긋나지 않는다.
- 현재 단계에서는 "완전한 동시성 안전성"보다 "구현 가능성과 설명 가능성"이 더 중요한 성공 조건이다.
- 팀은 `pthread`, socket, semaphore 기반 서버 구조를 구현하고 디버깅할 수 있다.
- 읽기 요청 무잠금 정책에서 생길 수 있는 race 가능성은 과제 발표에서 기술 부채로 솔직하게 설명해도 된다.
- 결과적으로 이 프로젝트의 진짜 가치는 기능 추가보다 "기존 DB 엔진을 감싸는 서버 경계 설계"에 있다.

## Approaches Considered

### Approach A: 얇은 서버 래퍼 + 중앙 write mutex

기존 엔진 위에 `server.c`, `thread_pool.c/h`, `api.c/h`, `db_server.c/h`만 얹는 구조다.

- 장점
- 현재 명세서와 가장 잘 맞는다.
- 구현 범위가 선명하다.
- 발표용 아키텍처 설명이 쉽다.
- 기존 코드 변경을 최소화할 수 있다.
- 단점
- 읽기/쓰기 혼합 부하에서 정합성 한계가 남는다.
- 기능이 늘어나면 `db_server`에 책임이 몰릴 수 있다.

### Approach B: 처음부터 read/write 확장성을 염두에 둔 서버 계층

초기부터 reader-writer lock, 더 일반적인 라우팅/실행기 추상화, 다중 테이블 가능성을 고려한 구조로 짜는 방식이다.

- 장점
- 장기적으로는 더 깔끔한 확장이 가능하다.
- 이후 `UPDATE`, `DELETE`, 다중 테이블, 디스크 저장으로 가기 편하다.
- 단점
- 이번 MVP 범위를 과하게 넓힌다.
- 구현량과 디버깅 비용이 급격히 커진다.
- 팀 발표에서 "왜 이렇게 복잡해졌는지" 설명 비용이 높다.

## Recommended Approach

Approach A를 권장한다. 이 프로젝트는 지금 "DB 제품화"보다 "기존 엔진을 안전한 서버 형태로 감싸는 첫 번째 경계"를 만드는 단계에 가깝다. 그래서 설계의 우선순위는 확장성의 이상형보다 다음 세 가지여야 한다.

1. 기존 엔진 재사용이 명확할 것
2. 동시성 정책이 한 문장으로 설명될 것
3. 테스트와 데모 시나리오가 구현 범위와 1:1로 대응될 것

권장 구현 흐름은 아래와 같다.

1. `Makefile`에 `server` 빌드 타깃과 `-pthread` 링크를 추가한다.
2. `db_server`에서 공유 `Table`, write mutex, timeout 기반 SQL 실행 wrapper를 캡슐화한다.
3. `thread_pool`에서 bounded queue와 worker lifecycle을 고정 크기로 구현한다.
4. `api` 레이어에서 `/health`, `/metrics`, `/query`와 JSON envelope를 책임지게 한다.
5. `server.c`는 accept, 최소 HTTP 파싱, enqueue, 즉시 에러 응답 같은 입구 역할만 담당한다.

이렇게 나누면 "네트워크", "큐/스레드", "DB 실행", "응답 포맷"의 경계가 살아 있고, 어느 한 파일도 과도하게 비대해지지 않는다.

## Open Questions

- `SELECT` 무잠금 정책에서 실제로 허용 가능한 race 범위를 어디까지 팀이 감수할 것인가?
- write mutex timeout 구현 시 busy wait + sleep 간격을 몇 ms로 둘 것인가?
- malformed HTTP request 판정 범위를 어디까지 처리할 것인가? 최소 파싱만 할지, header validation도 더 할지 결정이 필요하다.
- `requestId` 생성 방식은 단순 증가 정수로 충분한가, 아니면 thread-safe한 문자열 생성기를 둘 것인가?
- 긴 body를 읽다가 `4096` bytes를 넘는 경우를 언제 잘라내고 언제 `413`을 응답할지 입구 정책을 명확히 해야 한다.
- 부분 write, client disconnect, shutdown 시 worker 정리 같은 운영성 이슈를 이번 범위에 얼마나 포함할지 정해야 한다.

## Success Criteria

- 클라이언트가 `POST /query`로 SQL을 보내고 JSON 응답을 안정적으로 받을 수 있다.
- `GET /health`와 `GET /metrics`가 명세된 형태로 동작한다.
- 기존 `INSERT`, `SELECT`, `WHERE id` 조회 기능이 깨지지 않는다.
- 동시 `INSERT` 이후에도 `id`, `next_id`, `pk_index` 불일치가 발생하지 않는다.
- 큐 포화 시 `503 queue_full`, lock timeout 시 `408 lock_timeout`이 명시적으로 반환된다.
- 테스트와 데모에서 "무엇이 보장되고 무엇이 아직 보장되지 않는지"를 설명할 수 있다.

## Distribution Plan

- 1차 배포 단위는 실행 중인 퍼블릭 서비스가 아니라 소스 코드 저장소와 실행 가능한 `server` 바이너리다.
- 기본 배포 채널은 팀 GitHub 저장소다. 코드, README, curl 예시, 테스트 스크립트를 함께 관리하는 것이 가장 자연스럽다.
- 가능하면 GitHub Actions에서 `unit_test`와 API 테스트를 같이 돌리면 신뢰도가 올라간다.
- 발표/시연은 로컬 또는 팀 환경에서 `./server --port 8080 --workers 4 --queue-size 64 --timeout-ms 3000`로 실행하고 `curl`로 검증하는 흐름이 가장 적합하다.
- 과제/팀 정책이 불명확하면 공개 저장소보다 private 저장소 또는 팀 저장소에 먼저 올리는 편이 안전하다.

## Next Steps

1. `Makefile`과 기본 빌드 경로를 정리해 `server` 타깃이 생성되도록 한다.
2. `db_server.c/h`를 먼저 구현해 공유 상태와 write mutex 경계를 확정한다.
3. `thread_pool.c/h`에 bounded queue, semaphore wakeup, worker 종료 경로를 넣는다.
4. `api.c/h`와 `server.c`를 붙여 `/health`, `/metrics`, `/query`를 end-to-end로 연결한다.
5. API 통합 테스트와 동시성 테스트를 추가해 `queue_full`, `lock_timeout`, 다중 `INSERT`, 다중 `SELECT`를 검증한다.
6. README에 실행법, 요청 예시, 현재 한계, 향후 개선 방향을 문서화한다.

## What I noticed about how you think

- "새 DBMS를 처음부터 확장하는 것"이 아니라 "이미 구현된 엔진을 서버 환경에서 안전하게 호출"하는 데 집중하고 있어서, 범위를 잘 자르는 감각이 분명하다.
- "단순하고 설명 가능한 동시성 모델"을 채택한다고 못 박은 점을 보면, 구현 난이도보다 팀이 발표와 유지보수까지 감당할 수 있는 구조를 더 중시하고 있다.
- `queue_full`, `lock_timeout`, `usedIndex`, `metrics`까지 명세에 넣은 걸 보면 기능 구현만이 아니라 운영 관찰 가능성까지 챙기려는 성향이 있다.
- 읽기 lock 부재를 숨기지 않고 "현재 MVP의 한계와 향후 개선점"으로 드러낸 점이 좋다. 이 프로젝트에서 중요한 건 완벽한 척하는 설계보다, 어디까지가 의도된 타협인지 명확히 남기는 것이다.
