## Problem Statement

이 프로젝트의 겉모습은 "HTTP API 서버를 추가한다"이지만, 실제 문제는 그보다 좁고 더 까다롭다. 지금 저장소는 이미 동작하는 REPL 기반 SQL 엔진을 갖고 있다. `main.c`는 입력을 받고 `SQLResult`를 출력하는 얇은 셸이고, 실제 상태와 실행 규칙은 `sql.c`, `table.c`, `bptree.c`에 들어 있다. 그래서 이번 작업의 핵심은 새 DBMS를 설계하는 것이 아니라, 이 단일 사용자 엔진을 여러 HTTP 클라이언트가 호출할 수 있는 서비스 경계로 감싸는 것이다.

즉 우리가 풀어야 할 문제는 단순히 `POST /query`를 만드는 것이 아니다. 더 정확히는, 이미 있는 엔진의 동작을 가능한 한 유지한 채로 요청 큐, 워커 스레드, JSON 계약, 오류 매핑, metrics, timeout, 그리고 명시적인 동시성 정책을 추가하는 것이다. 서버를 붙이는 순간 새로 생기는 책임은 네트워크 자체보다 "공유 상태를 어디까지 안전하다고 말할 수 있는가"에 있다.

이제 그 질문에 대한 답은 명확하다. 이번 설계는 `SELECT lock-free`를 포기하고, 단일 `pthread_rwlock_t`를 엔진 경계에 도입한다. `SELECT`는 read lock 아래에서만 `rows`와 B+Tree를 읽고, `INSERT`는 write lock 아래에서만 `next_id`, `rows`, `pk_index`, `leaf->next`를 바꾼다. 이렇게 해야만 현재 엔진 구조를 크게 바꾸지 않으면서도 "읽기 병렬"을 정직하게 약속할 수 있다. 이 설계의 핵심은 HTTP를 붙이는 것이 아니라, 기존 엔진에 대해 안전한 동시성 계약을 하나 추가하는 것이다.

## What Makes This Cool

- 기존 엔진을 버리지 않고 `sql_execute()`와 `SQLResult`를 그대로 재사용한다.
- `client -> queue -> thread pool -> rwlock -> sql_execute -> table/B+Tree`라는 흐름이 눈에 보여서 발표와 데모에 강하다.
- 단순 CRUD 서버가 아니라 `queue_full`, `lock_timeout`, `usedIndex`, `metrics` 같은 운영 시그널을 API 계약에 드러낸다.
- "읽기 병렬"을 더 이상 연출이 아니라 실제 약속으로 말할 수 있다.

## Constraints

- 기존 엔진의 public surface는 최대한 유지한다. 핵심 재사용 지점은 `sql_execute(Table *, const char *)`와 `SQLResult`다.
- 지원 SQL은 현재 파서가 처리하는 범위로 제한한다. 이번 단계에서 DDL, 다중 테이블, 영속화는 하지 않는다.
- 외부 JSON 라이브러리는 도입하지 않는다.
- 엔드포인트는 `GET /health`, `GET /metrics`, `POST /query`만 제공한다.
- 빈 조회는 API에서 성공으로 다뤄야 한다. 현재 `sql.c`는 조회 결과가 없으면 `SQL_STATUS_NOT_FOUND`를 줄 수 있으므로, 서버 계층에서 `200 + rows: []`로 변환하는 규칙이 필요하다.
- DB 락은 `db_server` 경계에만 둔다. `sql.c`, `table.c`, `bptree.c`는 lock-unaware 상태를 유지한다.
- 이번 범위에서는 `UPDATE`, `DELETE`가 없으므로, 한 번 삽입된 `Record`는 shutdown 전까지 수정·해제되지 않는다고 본다.
- 팀원이 화이트보드에서 2분 안에 설명 못 하는 동시성 규칙은 이 MVP에 과하다. 락 모델은 단순해야 한다.

## Premises

1. 엔진 경계는 이미 충분히 좋다. `main.c`를 HTTP 서버로 바꾸는 게 맞지, 파서와 실행기를 다시 짜는 게 맞지 않다.
2. `SQL_STATUS_NOT_FOUND`는 DB 기능의 결핍이 아니라 API 매핑 문제다. 이건 `db_server` 또는 API 계층에서 해결해야 한다.
3. 현재 엔진에서 위험한 것은 `Record` 내용보다 `rows`, `size`, `pk_index`, leaf chain 같은 공유 컨테이너와 인덱스 구조다. 이 부분은 동기화 없이 읽고 쓰면 안 된다.
4. `pthread_rwlock_t`를 `db_server` 경계에 두는 것이, 현재 코드 변경량을 크게 늘리지 않으면서 mixed read/write UB를 피하고 read parallelism을 유지하는 가장 작은 정직한 해결책이다.
5. `usedIndex`는 발표용 필드로는 좋다. 다만 핸들러마다 느슨한 문자열 비교로 계산하면 금방 거짓말이 된다. 파서와 같은 규칙으로 한 곳에서 계산해야 한다.

## Approaches Considered

### Approach A: 명세 준수 lock-free SELECT

요약: 현재 명세를 거의 그대로 따른다. `INSERT`만 write mutex로 직렬화하고 `SELECT`는 별도 table lock 없이 실행한다. mixed read/write는 MVP 한계라고 문서와 데모에서 밝힌다.

노력도: M

장점:

- 현재 명세와 발표 흐름에 가장 잘 맞는다.
- 구현 diff가 비교적 작다.
- "읽기는 병렬, 쓰기는 직렬"이라는 메시지가 직관적이다.

단점:

- 현재 엔진 구조에서는 mixed read/write가 실제로 unsafe 하다.
- happy path 테스트가 통과해도 안전성을 증명하지 못한다.
- 시연 중 압박이 걸릴 때 가장 터지기 쉬운 지점이 바로 이 부분이다.

적합한 경우: 채점 기준이 lock-free SELECT를 하드하게 요구할 때만.

### Approach B: 전역 mutex 직렬화

요약: fixed thread pool, bounded queue, metrics, HTTP API는 그대로 구현하되, `sql_execute()` 진입을 모두 하나의 mutex로 감싼다. 네트워크는 병렬이지만 인메모리 엔진은 단일 실행 경로를 유지한다.

노력도: M

장점:

- 정확성 설명이 가장 깔끔하다.
- 구현과 테스트가 가장 단순하다.
- 현재 엔진을 concurrent reader/writer로 다루며 생기는 undefined behavior를 빠르게 없앨 수 있다.

단점:

- 엔진 내부 레벨에서 "읽기 병렬"을 주장할 수 없다.
- thread pool이 있어도 핵심 데모 메시지가 약해진다.

적합한 경우: 데모 신뢰성과 완주 가능성이 최우선일 때.

### Approach C: read/write lock 기반 경계

요약: `db_server`에 단일 `pthread_rwlock_t`를 두고, `SELECT`는 read lock, `INSERT`는 write lock을 거쳐서 엔진으로 진입하게 만든다. thread pool과 API 구조는 그대로 유지한다.

노력도: L

장점:

- "읽기 병렬"을 정직하게 말할 수 있다.
- mixed read/write UB를 피하면서도 기존 엔진을 대부분 재사용할 수 있다.
- 이후 `UPDATE`, `DELETE`, 추가 테이블로 가는 구조가 가장 자연스럽다.

단점:

- 전역 mutex보다 구현과 테스트가 더 복잡하다.
- 락 범위와 락 해제 시점을 명확하게 못 박아야 한다.
- 잘못 구현하면 read parallelism은 얻지 못하고 복잡도만 늘어난다.

적합한 경우: 이번 프로젝트처럼 "읽기 병렬" 메시지도 살리고 싶고, 동시에 안전성도 실제로 확보하고 싶을 때.

## Recommended Approach

이번 결정은 Approach C, 즉 read/write lock 기반 경계다. 이건 앞서 설명한 세 가지 해결책 중 두 번째, "read/write lock 방식"에 해당한다.

이 선택의 이유는 단순하다. 지금 이 프로젝트가 제일 먼저 증명해야 하는 건 "여러 요청을 받아도 기존 엔진을 거짓말 없이 감쌀 수 있다"는 점이고, 동시에 발표와 데모에서는 "읽기 요청은 병렬로 처리된다"는 메시지도 살리고 싶다. 전역 mutex는 너무 안전하지만 메시지가 약하고, lock-free SELECT는 메시지는 화려하지만 현재 코드 구조 기준으로 정직하지 않다. rwlock은 그 둘 사이에서 가장 좋은 균형점이다.

## Read/Write Lock Rules

1. 모든 DB 엔진 접근은 `db_server_execute()` 같은 단일 진입점으로만 들어간다.
2. `SELECT`는 `sql_execute()` 호출 전 read lock을 획득하고, 결과 포인터 배열과 row count가 완전히 만들어질 때까지 유지한다.
3. `INSERT`는 `sql_execute()` 호출 전 write lock을 획득하고, `next_id`, `rows`, `size`, `capacity`, `pk_index`, B+Tree split 경로가 모두 끝난 뒤에 해제한다.
4. `sql.c`, `table.c`, `bptree.c` 내부에는 락 코드를 넣지 않는다. 락은 서버 경계에서만 관리한다.
5. HTTP 파싱, JSON 파싱, 요청 검증은 DB 락을 잡기 전에 끝낸다.
6. `health`와 `metrics`는 DB 구조를 직접 읽지 않는 한 rwlock을 잡지 않는다.
7. queue mutex, metrics mutex, socket write와 DB rwlock을 중첩해서 오래 잡지 않는다. 특히 "queue lock을 잡은 채로 DB lock 대기"는 금지한다.
8. 나중에 `UPDATE`나 `DELETE`가 생기면 기본값은 write lock 경로다.
9. "lock-free SELECT"를 더 이상 주장하지 않는다. 이번 v1의 약속은 "read-safe parallel SELECT"다.

## Four Fixed Contracts Before Coding

### 1. 결과 포인터 수명과 unlock 시점은 고정한다

- 현재 `SQLResult`는 `Record` 복사본이 아니라 table-owned `Record *`를 담는다.
- 따라서 `SELECT` 요청은 `sql_execute()`가 끝났다고 바로 read lock을 풀지 않는다.
- 서버는 응답 JSON 문자열을 끝까지 만든 뒤에만 read lock을 해제한다.
- read lock을 풀어 놓고 나서 `result.records[index]->...`를 다시 읽는 코드는 금지한다.
- shutdown 순서는 `accept 중단 -> queue 종료 -> worker drain/join -> Table destroy -> rwlock destroy`로 고정한다.
- 직관적으로 말하면, 지금 구조는 "책 내용을 복사해 주는 구조"가 아니라 "책장 안의 책을 직접 펼쳐 보는 구조"에 가깝다. 응답에 필요한 내용을 다 써 내려가기 전에는 책장을 치우면 안 된다.

### 2. timeout은 lock 시간만이 아니라 요청 전체 시간을 기준으로 본다

- timeout budget은 요청이 서버에 받아들여져 queue에 들어가는 순간부터 시작한다.
- queue 대기 시간, worker 대기 시간, DB lock 대기 시간은 모두 같은 timeout budget을 공유한다.
- `queue_full`은 queue에 넣기 전에 거절된 경우에만 사용한다.
- queue에서 꺼냈을 때 이미 budget이 소진되었으면, worker는 `sql_execute()`에 들어가지 않고 바로 `lock_timeout`을 반환한다.
- rwlock 대기 중에 남은 budget이 소진되어도 같은 방식으로 `lock_timeout`을 반환한다.
- 직관적으로 말하면, 사용자가 체감한 기다림은 "락 앞에서 기다린 시간"만이 아니라 "서버 안에서 줄 서 있던 전체 시간"이다. 이번 명세는 그 전체 시간을 한 번에 센다.

### 3. lock 종류와 `usedIndex`는 같은 판단에서 나온다

- 서버는 실행 전에 요청을 한 번만 분류하고, 그 결과로 lock 종류와 `usedIndex`를 함께 결정한다.
- 이 분류는 `db_server_classify_query()` 같은 단일 함수에서 관리하고, API 핸들러의 ad-hoc 문자열 비교로 계산하지 않는다.
- v1 기준으로 `WHERE id ...` 계열 조회는 index 경로로 보고 `usedIndex = true`로 기록한다.
- `SELECT * FROM users`, `WHERE name = ...`, `WHERE age ...` 경로는 scan 경로로 보고 `usedIndex = false`로 기록한다.
- `INSERT`는 write 경로로 분류하고 `usedIndex`는 응답 필드에서 생략하거나 `false`로 통일한다. 구현 중 하나를 선택하되 문서와 코드가 같은 규칙을 따라야 한다.
- 직관적으로 말하면, "어떻게 실행했는지"와 "무슨 경로를 썼다고 API가 말하는지"가 서로 다르면 안 된다. 실행 판단과 발표용 필드는 같은 스위치에서 나와야 한다.

### 4. HTTP/JSON 계약은 구현 전에 고정한다

- `POST /query` 요청 body는 `{"query":"SELECT * FROM users;"}` 형태로 고정한다.
- `POST /query` 성공 응답은 항상 하나의 envelope를 사용한다. 예: `{"ok":true,"action":"select","usedIndex":false,"rowCount":0,"rows":[]}`
- `INSERT` 성공 응답도 같은 envelope를 유지한다. 예: `{"ok":true,"action":"insert","insertedId":1}`
- 빈 조회는 HTTP 200과 `ok: true`, `rowCount: 0`, `rows: []`로 반환한다.
- 오류 응답도 envelope를 통일한다. 예: `{"ok":false,"error":{"type":"syntax_error","message":"..."}}`
- `EXIT`와 `QUIT`는 REPL 전용 명령으로 취급하고, API 서버에서는 프로세스를 종료하지 않는다. 이 입력은 HTTP 400 계열 오류 응답으로 거절한다.
- `GET /health`는 최소한 `{"ok":true,"status":"ok"}`를 반환하고, `GET /metrics`는 문서에 적힌 운영 지표만 JSON으로 노출한다.
- 직관적으로 말하면, 같은 질문에는 항상 같은 모양의 답이 나와야 한다. 그래야 테스트, curl 예시, 발표 메시지, 실제 구현이 모두 같은 방향으로 정렬된다.

## Open Questions

- 타깃 환경에서 `pthread_rwlock_t`와 관련 timed wait API를 어디까지 사용할 것인가?
- malformed HTTP request의 판정 범위를 어디까지 둘 것인가? request line만 볼지, header/body framing까지 볼지 정해야 한다.
- client disconnect, partial write 같은 운영 edge case를 이번 범위에 넣을 것인가, 명시적 비범위로 둘 것인가?

## Success Criteria

- `GET /health`, `GET /metrics`, `POST /query`가 end-to-end로 동작한다.
- 기존 `unit_test`는 깨지지 않는다.
- 빈 `SELECT`는 HTTP 200과 `rows: []`로 반환된다.
- 기존 `INSERT`, `SELECT`, `WHERE id` 경로의 의미가 바뀌지 않는다.
- 여러 `SELECT` 요청이 동시에 성공하고, `INSERT`는 write lock 아래에서 안전하게 직렬화된다.
- mixed read/write에서 "락 없이 읽는다"는 전제를 버리고, rwlock 기반 안전성 규칙을 문서와 코드가 같은 방식으로 설명한다.
- `queue_full`, `lock_timeout`, syntax/query/internal error가 일관된 JSON envelope로 내려간다.

## Distribution Plan

- 1차 전달물은 저장소 안에서 빌드 가능한 `server` 바이너리와 README의 `curl` 예시다.
- 가장 가벼운 배포 경로는 GitHub 저장소 + 실행 방법 + smoke test 스크립트다.
- 시간이 허락하면 `unit_test`와 `server` 빌드를 확인하는 작은 CI를 붙이면 신뢰도가 많이 올라간다.
- 이건 로컬/과제 시연용 서비스다. TLS, auth, 인터넷 배포 같은 문제를 억지로 끌고 오지 않는 게 맞다.

## The Assignment

소켓 코드 한 줄 쓰기 전에, 이 프로젝트의 동시성 약속을 세 문장으로 먼저 고정해라.

- `SELECT`는 read lock 아래에서만 엔진을 읽는다.
- `INSERT`는 write lock 아래에서만 엔진을 변경한다.
- `sql.c`, `table.c`, `bptree.c`는 lock-unaware로 두고, 서버 경계에서만 동기화한다.

이 세 문장이 먼저 맑아지면, 이후 API 계약, 테스트, 발표 메시지, README가 전부 같은 방향으로 정렬된다.
