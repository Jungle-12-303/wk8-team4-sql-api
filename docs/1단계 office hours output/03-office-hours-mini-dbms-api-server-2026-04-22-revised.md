## Problem Statement

이 프로젝트의 겉모습은 "HTTP API 서버를 추가한다"이지만, 실제 문제는 그보다 좁고 더 까다롭다. 지금 저장소는 이미 동작하는 REPL 기반 SQL 엔진을 갖고 있다. `main.c`는 입력을 받고 `SQLResult`를 출력하는 얇은 셸이고, 실제 상태와 실행 규칙은 `sql.c`, `table.c`, `bptree.c`에 들어 있다. 그래서 이번 작업의 핵심은 새 DBMS를 설계하는 것이 아니라, 이 단일 사용자 엔진을 여러 HTTP 클라이언트가 호출할 수 있는 서비스 경계로 감싸는 것이다.

즉 우리가 풀어야 할 문제는 단순히 `POST /query`를 만드는 것이 아니다. 더 정확히는, 이미 있는 엔진의 동작을 가능한 한 유지한 채로 요청 큐, 워커 스레드, JSON 계약, 오류 매핑, metrics, timeout, 그리고 무엇보다 명시적인 동시성 정책을 추가하는 것이다. 서버를 붙이는 순간 새로 생기는 책임은 네트워크 자체보다 "공유 상태를 어디까지 안전하다고 말할 수 있는가"에 있다.

여기서 가장 위험한 함정은 `SELECT`를 table lock 없이 실행하겠다는 전제를 너무 가볍게 보는 것이다. 현재 읽기 경로는 `table_collect_all()`로 `rows` 포인터 배열을 읽고, `table_find_by_id_condition()`으로 leaf chain을 따라간다. 반대편 쓰기 경로는 `table_insert()`와 `bptree_split_leaf_and_insert()`를 통해 `rows`, `next_id`, `pk_index`, `leaf->next`를 바꾼다. 이건 단순한 stale read 문제가 아니라 mixed read/write에서 C 메모리 모델 기준 undefined behavior로 이어질 수 있다. 그래서 이 설계의 첫 질문은 "API를 어떻게 붙일까?"가 아니라 "이 엔진에 대해 무엇을 안전하다고 약속할 수 있고, 무엇은 아직 약속하지 않는다고 명확히 적을까?"여야 한다.

## What Makes This Cool

- 기존 엔진을 버리지 않고 `sql_execute()`와 `SQLResult`를 그대로 재사용한다.
- `client -> queue -> thread pool -> sql_execute -> table/B+Tree`라는 흐름이 눈에 보여서 발표와 데모에 강하다.
- 단순 CRUD 서버가 아니라 `queue_full`, `lock_timeout`, `usedIndex`, `metrics` 같은 운영 시그널을 API 계약에 드러낸다.
- "새 서버를 만들었다"보다 "학습용 엔진을 정직한 서비스 경계로 감쌌다"는 점이 더 멋지다.

## Constraints

- 기존 엔진의 public surface는 최대한 유지한다. 핵심 재사용 지점은 `sql_execute(Table *, const char *)`와 `SQLResult`다.
- 지원 SQL은 현재 파서가 처리하는 범위로 제한한다. 이번 단계에서 DDL, 다중 테이블, 영속화는 하지 않는다.
- 외부 JSON 라이브러리는 도입하지 않는다.
- 엔드포인트는 `GET /health`, `GET /metrics`, `POST /query`만 제공한다.
- 빈 조회는 API에서 성공으로 다뤄야 한다. 현재 `sql.c`는 조회 결과가 없으면 `SQL_STATUS_NOT_FOUND`를 줄 수 있으므로, 서버 계층에서 `200 + rows: []`로 변환하는 규칙이 필요하다.
- 이번 프로젝트의 가치는 "완전한 DB 수준 격리"보다 "기존 엔진을 설명 가능한 서버로 감싼 것"에 있다.
- 팀원이 화이트보드에서 2분 안에 설명 못 하는 동시성 규칙은 이 MVP에 과하다.

## Premises

1. 엔진 경계는 이미 충분히 좋다. `main.c`를 HTTP 서버로 바꾸는 게 맞지, 파서와 실행기를 다시 짜는 게 맞지 않다.
2. `SQL_STATUS_NOT_FOUND`는 DB 기능의 결핍이 아니라 API 매핑 문제다. 이건 `db_server` 또는 API 계층에서 해결해야 한다.
3. 현재 명세의 `SELECT lock-free`는 유일하게 강하게 의심되는 전제다. 이걸 유지하려면 mixed read/write 안전성을 보장한다고 말하면 안 된다.
4. `usedIndex`는 발표용 필드로는 좋다. 다만 핸들러마다 느슨한 문자열 비교로 계산하면 금방 거짓말이 된다. 파서와 같은 규칙으로 한 곳에서 계산해야 한다.
5. 이 프로젝트의 진짜 가치는 "기존 SQL 엔진을 서버 경계로 확장했다"는 데 있다. 동시성에서 괜히 영웅 서사를 만들 필요는 없다.

## Approaches Considered

### Approach A: 명세 준수 MVP

요약: 현재 명세를 거의 그대로 따른다. `server.c`, `thread_pool.c/h`, `api.c/h`, `db_server.c/h`를 추가하고, `INSERT`만 write mutex로 직렬화하며 `SELECT`는 별도 table lock 없이 실행한다. 대신 mixed read/write는 MVP 한계라고 문서와 데모에서 정직하게 밝힌다.

노력도: M

장점:

- 현재 명세와 발표 흐름에 가장 잘 맞는다.
- 구현 diff가 비교적 작다.
- "읽기는 병렬, 쓰기는 직렬"이라는 메시지가 직관적이다.

단점:

- 현재 엔진 구조에서는 mixed read/write가 실제로 unsafe 하다.
- 테스트가 happy path를 보여줄 수는 있어도 undefined behavior를 없애주지는 못한다.
- 시연 중 압박이 걸릴 때 가장 터지기 쉬운 지점이 바로 이 부분이다.

적합한 경우: 과제나 발표 기준이 이 동시성 서사를 사실상 요구할 때.

### Approach B: 정확성 우선 MVP

요약: fixed thread pool, bounded queue, metrics, HTTP API는 그대로 구현하되, `sql_execute()` 진입은 전부 하나의 DB mutex로 감싼다. 네트워크와 요청 처리는 병렬이지만 인메모리 엔진은 단일 실행 경로를 유지한다.

노력도: M

장점:

- 정확성 설명이 가장 깔끔하다.
- 구현과 테스트가 가장 단순하다.
- 현재 엔진을 concurrent reader/writer로 다루며 생기는 undefined behavior를 피할 수 있다.

단점:

- 엔진 내부 레벨에서 "읽기 병렬"을 주장할 수 없다.
- 발표 관점에서는 덜 화려하다.

적합한 경우: 데모 신뢰성과 완주 가능성이 병렬 읽기 연출보다 중요할 때.

### Approach C: 장기 구조를 먼저 여는 설계

요약: `pthread_rwlock_t` 같은 read/write 보호 계층을 도입한다. `SELECT`는 read lock, `INSERT`는 write lock을 거쳐서 진입하게 만들고, thread pool과 API 구조는 그대로 유지한다.

노력도: L

장점:

- "읽기 병렬"을 정직하게 말할 수 있다.
- 이후 `UPDATE`, `DELETE`, 추가 테이블로 가는 구조가 가장 좋다.
- 서버와 엔진 사이의 계약이 더 명확해진다.

단점:

- 이번 범위 대비 구현량이 커진다.
- 테스트 매트릭스가 급격히 넓어진다.
- 팀 설명 비용도 같이 올라간다.

적합한 경우: 이 저장소를 과제 이후에도 계속 발전시킬 계획이 분명할 때.

## Recommended Approach

내 추천은 Approach B다. 지금 이 프로젝트가 제일 먼저 증명해야 하는 건 "여러 요청을 받아도 기존 엔진을 거짓말 없이 감쌀 수 있다"는 점이지, "읽기를 최대한 병렬로 보이게 만들었다"는 점이 아니다.

만약 채점 기준이 문자 그대로 "`SELECT`는 별도 table lock 없이 실행"을 요구한다면 Approach A로 가도 된다. 하지만 그 경우 문서의 한 문장을 반드시 고쳐야 한다. "안전하다"가 아니라 "mixed read/write 안전성은 이번 MVP에서 보장하지 않는다"라고 써야 한다. 서버 경계에서 가장 나쁜 선택은 애매한 문장이다. 코드보다 먼저 약속이 흐려진다.

## Open Questions

- `SELECT without table lock`은 과제의 하드 요구사항인가, 아니면 팀이 스스로 넣은 stretch goal인가?
- `requestId`는 thread-safe하게 어떻게 만들 것인가? atomic counter면 충분한가?
- `usedIndex`는 정확히 어떤 규칙으로 계산할 것인가? 핸들러별 ad-hoc 문자열 비교는 피해야 한다.
- insert 도중 메모리 할당 실패가 나면 `next_id` gap을 허용할 것인가?
- malformed HTTP request의 판정 범위를 어디까지 둘 것인가? request line만 볼지, header/body framing까지 볼지 정해야 한다.
- worker shutdown, client disconnect, partial write 같은 운영 edge case를 이번 범위에 넣을 것인가, 명시적 비범위로 둘 것인가?

## Success Criteria

- `GET /health`, `GET /metrics`, `POST /query`가 end-to-end로 동작한다.
- 기존 `unit_test`는 깨지지 않는다.
- 빈 `SELECT`는 HTTP 200과 `rows: []`로 반환된다.
- 기존 `INSERT`, `SELECT`, `WHERE id` 경로의 의미가 바뀌지 않는다.
- `queue_full`, `lock_timeout`, syntax/query/internal error가 일관된 JSON envelope로 내려간다.
- 팀이 최종 동시성 모델을 2분 안에 손으로 설명할 수 있다.
- 최종 문서와 데모가 "무엇이 안전하고, 무엇이 아직 아니다"를 숨기지 않는다.

## Distribution Plan

- 1차 전달물은 저장소 안에서 빌드 가능한 `server` 바이너리와 README의 `curl` 예시다.
- 가장 가벼운 배포 경로는 GitHub 저장소 + 실행 방법 + smoke test 스크립트다.
- 시간이 허락하면 `unit_test`와 `server` 빌드를 확인하는 작은 CI를 붙이면 신뢰도가 많이 올라간다.
- 이건 로컬/과제 시연용 서비스다. TLS, auth, 인터넷 배포 같은 문제를 억지로 끌고 오지 않는 게 맞다.

## Next Steps

1. `Makefile`에 `server` target과 필요한 compile/link option을 먼저 추가한다.
2. `db_server_execute()`를 만든다. 이 함수가 concurrency policy, timeout, `SQLResult -> API result` 매핑의 중심이 된다.
3. 그 다음 `api.c/h`에서 JSON envelope, `rows: []`, JSON escaping을 먼저 확정한다.
4. 이후 request queue와 worker pool을 붙인다.
5. happy path smoke test 하나, concurrency policy test 하나를 만든다.
6. README는 마지막에 갱신한다. 구현이 끝나기 전 문서부터 정교하게 쓰면 보통 코드가 문서에 끌려간다.

## The Assignment

소켓 코드 한 줄 쓰기 전에, 이 프로젝트의 약속을 두 문장으로 먼저 고정해라.

- 우리가 보장하는 것 1문장
- 우리가 아직 보장하지 않는 것 1문장

예시는 이렇다.

- "We provide bounded multi-client HTTP access to the existing in-memory SQL engine."
- "We do not claim mixed read/write isolation safety in v1 unless we serialize engine access."

이 두 문장이 먼저 맑아지면, 이후 API 계약, 테스트, 발표 메시지, README가 전부 같은 방향으로 정렬된다.

## What I noticed about how you think

- "새 DBMS를 처음부터 확장하는 것이 아니라"라고 먼저 못 박은 건 아주 좋은 감각이다. 재사용 가능한 경계를 보고 있다.
- 비범위를 공격적으로 잘라냈다. DDL, persistence, TLS, auth를 이번 단계에서 안 하겠다고 분명히 적은 건 설계 감각이다.
- 어색한 지점을 숨기지 않았다. mixed read/write 한계를 이미 문서에서 인정하고 있다는 건 기술 부채를 솔직하게 다루는 태도다.
- 기능 목록보다 발표 흐름과 설명 가능성을 같이 설계하고 있다. 이건 학습 프로젝트에서 생각보다 큰 차이를 만든다.
