# 요청 요약

- CSV 저장/조회 책임을 실행기에서 분리해 최소 스토리지 인터페이스를 도입한다.
- 새 스토리지 API는 SQL 문장 구조체를 직접 받지 않고, 스키마와 정리된 데이터만 받는다.

# 결정 사항

- 스토리지 계층은 함수형 인터페이스 두 개만 공개한다.
- `INSERT`, `SELECT` 모두 스토리지 함수를 통과하도록 정리한다.
- 백엔드 확장용 함수 포인터 구조체나 핸들 타입은 도입하지 않는다.

# 현재 브랜치 상태

- `simplify` 브랜치에서 작업했다.

# 완료한 작업

- `include/sqlproc.h`에 `storage_append_row`, `storage_print_rows` 선언을 추가했다.
- `src/storage.c`를 새로 만들어 CSV 경로 계산, 헤더 생성/검증, 행 추가, 조회 출력을 옮겼다.
- `src/executor.c`는 스키마/컬럼/타입 검증과 스토리지 호출만 담당하도록 정리했다.
- `tests/test_runner.c`에 헤더 자동 생성, 데이터 파일 없는 `SELECT`, 헤더 불일치 회귀를 보강했다.
- `README.md`의 디렉터리 구조와 실행 흐름 설명을 현재 구조에 맞게 갱신했다.

# 리뷰 결과

- 스토리지 함수가 SQL 문장 구조체를 직접 받지 않고도 현재 기능 범위를 모두 처리하는 것을 확인했다.
- 실행기와 CSV 파일 입출력 코드가 분리되어 책임 경계가 이전보다 명확해졌다.
